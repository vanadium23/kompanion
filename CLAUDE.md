## Flow-Next

This project uses Flow-Next for task tracking. Use `.flow/bin/flowctl` instead of markdown TODOs or TodoWrite.

**Quick commands:**
```bash
.flow/bin/flowctl list                # List all epics + tasks
.flow/bin/flowctl epics               # List all epics
.flow/bin/flowctl tasks --epic fn-N   # List tasks for epic
.flow/bin/flowctl ready --epic fn-N   # What's ready
.flow/bin/flowctl show fn-N.M         # View task
.flow/bin/flowctl start fn-N.M        # Claim task
.flow/bin/flowctl done fn-N.M --summary-file s.md --evidence-json e.json
```

**Creating a spec** ("create a spec", "spec out X", "write a spec for X"):

A spec = an epic. Create one directly — do NOT use `/flow-next:plan` (that breaks specs into tasks).

```bash
.flow/bin/flowctl epic create --title "Short title" --json
.flow/bin/flowctl epic set-plan <epic-id> --file - --json <<'EOF'
# Title

## Goal & Context
Why this exists, what problem it solves.

## Architecture & Data Models
System design, data flow, key components.

## API Contracts
Endpoints, interfaces, input/output shapes.

## Edge Cases & Constraints
Failure modes, limits, performance requirements.

## Acceptance Criteria
- [ ] Testable criterion 1
- [ ] Testable criterion 2

## Boundaries
What's explicitly out of scope.

## Decision Context
Why this approach over alternatives.
EOF
```

After creating a spec, choose next step:
- `/flow-next:plan <epic-id>` — research + break into tasks
- `/flow-next:interview <epic-id>` — deep Q&A to refine the spec

**Rules:**
- Use `.flow/bin/flowctl` for ALL task tracking
- Do NOT create markdown TODOs or use TodoWrite
- Re-anchor (re-read spec + status) before every task

**More info:** `.flow/bin/flowctl --help` or read `.flow/usage.md`

## Conventions

## Naming Patterns
- Lowercase, single word: `auth`, `sync`, `storage`, `library`, `stats`
- Package names match directory names
- Lowercase with underscores: `book_postgres.go`, `progress_test.go`, `interfaces.go`
- Test files: `*_test.go` suffix (e.g., `auth_test.go`)
- Mock files: `mocks_test.go` (generated by mockgen)
- PascalCase for exported: `AuthService`, `BookShelf`, `ProgressRepo`
- camelCase for unexported: `response` (in `internal/controller/http/v1/error.go`)
- Interface names end with verb or noun: `Progress`, `Shelf`, `Storage`, `UserRepo`
- Repository interfaces: suffix `Repo` (e.g., `BookRepo`, `UserRepo`)
- PascalCase for exported: `NewBookShelf`, `RegisterUser`, `CheckPassword`
- camelCase for unexported: `hashPassword`, `comparePasswords`, `readPrefixedEnv`
- Constructor pattern: `New[Type]` (e.g., `NewConfig`, `NewBookShelf`, `NewMemoryUserRepo`)
- Factory functions in packages: `Init[Service]` (e.g., `InitAuthService`)
- camelCase: `bookID`, `sessionKey`, `hashedPassword`
- Short names in small scopes: `l` for logger, `pg` for postgres
- Error variables: `Err[Name]` (e.g., `ErrBookAlreadyExists`, `ErrAuth`)
- PascalCase or UPPER_SNAKE_CASE: `IncorrectPassword`, `UserNotFound`
## Code Style
- Standard `gofmt` formatting
- Imports grouped: standard library, external packages, internal packages
- golangci-lint (run via `make linter-golangci`)
- Package comments start with "Package [name]:"
- Exported functions have brief comments: `// New -.` style
- TODO comments format: `// TODO: description`
## Import Organization
- `pgx "github.com/jackc/pgx/v5"` for pgx driver
- `v1 "github.com/vanadium23/kompanion/internal/controller/http/v1"` for versioned packages
## Error Handling
## Logging
## Comments
- Start with "Package [name]" followed by brief description
- Example: `// Package app configures and runs application.`
- Brief one-liners with dash suffix: `// New -.`
- Explain purpose, not implementation
- Format: `// TODO: description`
- Found in code for known improvements (e.g., `// TODO: move session key to separate type`)
## Function Design
- Return entity structs by value: `entity.Book`
- Return errors as last return value
- Use named returns sparingly
## Module Design
- Public interfaces defined in `interface.go` or `interfaces.go`
- Implementations unexported, accessed via interface
- Interface: `internal/auth/interface.go` defines `AuthInterface`
- Implementation: `internal/auth/auth.go` implements `AuthService`
- Repository interfaces define data access: `BookRepo`, `UserRepo`, `ProgressRepo`
- Multiple implementations: memory (for testing), postgres (production)
## Configuration
- Prefix: `KOMPANION_` (e.g., `KOMPANION_PG_URL`, `KOMPANION_AUTH_USERNAME`)
- Read via `readPrefixedEnv()` helper
## HTTP Handlers
- Each route group in separate file: `users.go`, `sync.go`
- Constructor returns routes struct: `newSyncRoutes(handler *gin.RouterGroup, ...)`

## Architecture

## Pattern Overview
- Layered separation with interfaces defining boundaries between layers
- Dependency injection via constructors in `internal/app/app.go`
- Repository pattern for data access with multiple storage backends
- Interface-based abstractions enabling testability with mocks
- Domain entities in `internal/entity/` with no external dependencies
## Layers
- Purpose: Core domain models with business logic methods
- Location: `internal/entity/`
- Contains: `Book`, `Progress` structs with domain methods like `Filename()`, `MimeType()`
- Depends on: Only standard library and minimal deps (decimal for series indexing)
- Used by: All other layers
- Purpose: Business logic orchestration and domain operations
- Location: `internal/library/`, `internal/sync/`, `internal/stats/`, `internal/auth/`
- Contains: Service structs (`BookShelf`, `ProgressSyncUseCase`, `AuthService`) implementing domain interfaces
- Depends on: Entity layer, Repository interfaces, Storage interfaces
- Used by: Controller layer
- Purpose: Data persistence abstraction
- Location: `internal/library/book_postgres.go`, `internal/sync/progress_postgres.go`, `internal/auth/repo_*.go`
- Contains: PostgreSQL implementations of repository interfaces
- Depends on: `pkg/postgres`, Entity layer
- Used by: Service layer via interfaces
- Purpose: HTTP request handling and routing
- Location: `internal/controller/http/`
- Contains: Gin router groups and route handlers
- Depends on: Service layer interfaces, Logger, Auth
- Used by: HTTP server in `pkg/httpserver`
- Purpose: External concerns (database, HTTP server, logging, storage)
- Location: `pkg/`
- Contains: Configurable components that can be swapped
- Depends on: External packages (gin, pgx, zerolog)
- Used by: Application layer

- Session-based authentication stored in `auth_session` table
- Device credentials stored as MD5 hash (KOReader compatibility requirement)
