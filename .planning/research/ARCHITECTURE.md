# Architecture Patterns

**Domain:** Highlights Sync Feature for KOmpanion

## Recommended Architecture

The highlights sync feature should follow the existing Clean Architecture / Hexagonal-style pattern already established in KOmpanion. The architecture separates concerns into distinct layers with interfaces defining boundaries.

## Data Flow

### Highlight Sync Flow (KOReader to KOmpanion)

```
1. KOReader Exporter Plugin sends POST to /syncs/highlights
   |
   v
2. authDeviceMiddleware validates device credentials (x-auth-user, x-auth-key)
   |  - Extracts device_name into gin.Context
   v
3. highlightRoutes.syncHighlights() binds JSON request
   |  - Deserializes []entity.HighlightRequest
   v
4. HighlightSyncUseCase.Sync() processes each highlight
   |  - Sets AuthDeviceName from context
   |  - Validates required fields
   |  - Calls repo.Store() for each highlight
   v
5. HighlightDatabaseRepo.Store() persists to PostgreSQL
   |  - Uses UPSERT to avoid duplicates
   |  - Stores with timestamp for history
   v
6. Response: 200 OK with synced count
```

### Highlight Display Flow (Web UI)

```
1. User navigates to /books/{bookID}
   |
   v
2. booksRoutes.viewBook() fetches book details
   |  - Calls shelf.ViewBook()
   |  - Calls highlightRepo.GetByDocumentID()
   v
3. Template renders book.html with highlights data
   |  - Shows reading stats (existing)
   |  - Shows highlights list (new section)
```

## Proposed Data Model

### Entity: Highlight

```go
// internal/entity/highlight.go
package entity

type Highlight struct {
    ID             string    `json:"id"`               // UUID generated by KOmpanion
    DocumentID     string    `json:"document"`         // MD5 hash of book file (koreader_partial_md5)
    Text           string    `json:"text"`             // The highlighted text content
    Note           string    `json:"note,omitempty"`   // Optional user note
    Page           string    `json:"page"`             // Page number or location
    Chapter        string    `json:"chapter,omitempty"` // Optional chapter title
    Timestamp      int64     `json:"time"`             // Unix timestamp from KOReader
    Drawer         string    `json:"drawer,omitempty"` // Highlight style (underline, highlight, etc.)
    Color          string    `json:"color,omitempty"`  // Highlight color
    Device         string    `json:"device"`           // KOReader device name
    DeviceID       string    `json:"device_id"`        // KOReader device ID
    AuthDeviceName string    `json:"-"`                // Authenticated device (from middleware)
    CreatedAt      time.Time `json:"created_at"`       // Server timestamp
}
```

### Database Schema

```sql
-- migrations/YYYYMMDDHHMMSS_highlights.up.sql
CREATE TABLE highlight_annotations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    koreader_partial_md5 TEXT NOT NULL,

    -- Highlight content
    text TEXT NOT NULL,
    note TEXT,
    page TEXT NOT NULL,
    chapter TEXT,

    -- Highlight metadata
    drawer TEXT,
    color TEXT,

    -- Timestamps
    highlight_time TIMESTAMPTZ NOT NULL,  -- When user created highlight in KOReader
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- Device info
    koreader_device TEXT NOT NULL,
    koreader_device_id TEXT NOT NULL,
    auth_device_name TEXT NOT NULL,

    -- Deduplication hash (MD5 of text + page + time)
    highlight_hash TEXT NOT NULL
);

-- Indexes for efficient querying
CREATE INDEX idx_highlights_document ON highlight_annotations(koreader_partial_md5);
CREATE INDEX idx_highlights_time ON highlight_annotations(highlight_time DESC);
CREATE UNIQUE INDEX idx_highlights_unique ON highlight_annotations(highlight_hash);

COMMENT ON TABLE highlight_annotations IS 'Stores book highlights synced from KOReader';
COMMENT ON COLUMN highlight_annotations.highlight_hash IS 'MD5 hash for deduplication';
```

## Patterns to Follow

### Pattern 1: Repository Pattern with Interface

**What:** Define repository interface in service layer, implement in data layer.
**When:** All data access operations.
**Example:**

```go
// internal/highlight/interfaces.go
package highlight

import (
    "context"
    "github.com/vanadium23/kompanion/internal/entity"
)

//go:generate mockgen -source=interfaces.go -destination=./mocks_test.go -package=highlight_test

type HighlightRepo interface {
    Store(ctx context.Context, h entity.Highlight) error
    GetByDocumentID(ctx context.Context, documentID string) ([]entity.Highlight, error)
    GetByBookID(ctx context.Context, bookID string, limit int) ([]entity.Highlight, error)
}

type Highlight interface {
    Sync(ctx context.Context, documentID string, highlights []entity.Highlight) (int, error)
    Fetch(ctx context.Context, documentID string) ([]entity.Highlight, error)
}
```

### Pattern 2: Use Case / Service Layer

**What:** Encapsulate business logic in dedicated service structs.
**When:** All business operations beyond CRUD.
**Example:**

```go
// internal/highlight/sync.go
package highlight

import (
    "context"
    "crypto/md5"
    "encoding/hex"
    "fmt"
    "time"

    "github.com/vanadium23/kompanion/internal/entity"
)

type HighlightSyncUseCase struct {
    repo HighlightRepo
}

func NewHighlightSync(r HighlightRepo) *HighlightSyncUseCase {
    return &HighlightSyncUseCase{repo: r}
}

func (uc *HighlightSyncUseCase) Sync(ctx context.Context, documentID string, highlights []entity.Highlight) (int, error) {
    synced := 0
    for _, h := range highlights {
        h.DocumentID = documentID
        h.CreatedAt = time.Now()

        // Generate deduplication hash
        h.HighlightHash = generateHash(h.Text, h.Page, h.Timestamp)

        if err := uc.repo.Store(ctx, h); err != nil {
            // Log error but continue with other highlights
            continue
        }
        synced++
    }
    return synced, nil
}

func (uc *HighlightSyncUseCase) Fetch(ctx context.Context, documentID string) ([]entity.Highlight, error) {
    return uc.repo.GetByDocumentID(ctx, documentID)
}

func generateHash(text, page string, timestamp int64) string {
    data := fmt.Sprintf("%s:%s:%d", text, page, timestamp)
    hash := md5.Sum([]byte(data))
    return hex.EncodeToString(hash[:])
}
```

### Pattern 3: HTTP Route Handler

**What:** Bind JSON, call service, return appropriate response.
**When:** All HTTP endpoints.
**Example:**

```go
// internal/controller/http/v1/highlight.go
package v1

import (
    "net/http"

    "github.com/gin-gonic/gin"
    "github.com/vanadium23/kompanion/internal/entity"
    "github.com/vanadium23/kompanion/internal/highlight"
    "github.com/vanadium23/kompanion/pkg/logger"
)

type highlightRoutes struct {
    highlight highlight.Highlight
    l         logger.Interface
}

func newHighlightRoutes(handler *gin.RouterGroup, h highlight.Highlight, l logger.Interface) {
    r := &highlightRoutes{highlight: h, l: l}

    h := handler.Group("/")
    {
        h.POST("/highlights", r.syncHighlights)
        h.GET("/highlights/:document", r.fetchHighlights)
    }
}

type highlightSyncRequest struct {
    Document  string            `json:"document"`
    Title     string            `json:"title"`
    Author    string            `json:"author"`
    Highlights []entity.Highlight `json:"highlights"`
}

func (r *highlightRoutes) syncHighlights(c *gin.Context) {
    var req highlightSyncRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        r.l.Error(err)
        c.JSON(http.StatusBadRequest, gin.H{"message": "Bad request", "code": 4000})
        return
    }

    // Set auth device name for each highlight
    deviceName := c.GetString("device_name")
    for i := range req.Highlights {
        req.Highlights[i].AuthDeviceName = deviceName
        req.Highlights[i].Device = deviceName
    }

    synced, err := r.highlight.Sync(c, req.Document, req.Highlights)
    if err != nil {
        r.l.Error(err)
        c.JSON(http.StatusInternalServerError, gin.H{"message": "Internal server error", "code": 5000})
        return
    }

    c.JSON(http.StatusOK, gin.H{
        "synced": synced,
        "total":  len(req.Highlights),
    })
}

func (r *highlightRoutes) fetchHighlights(c *gin.Context) {
    documentID := c.Param("document")

    highlights, err := r.highlight.Fetch(c, documentID)
    if err != nil {
        r.l.Error(err)
        c.JSON(http.StatusInternalServerError, gin.H{"message": "Internal server error", "code": 5000})
        return
    }

    c.JSON(http.StatusOK, highlights)
}
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: Tight Coupling Between Layers

**What:** Direct database calls from controller layer.
**Why bad:** Makes testing difficult, violates separation of concerns.
**Instead:** Always use service interface, inject repository.

### Anti-Pattern 2: Duplicate Storage Without Deduplication

**What:** Storing same highlight multiple times on repeated sync.
**Why bad:** Wastes storage, confuses users with duplicate entries.
**Instead:** Use unique constraint on hash of (text + page + timestamp).

### Anti-Pattern 3: Ignoring Existing Auth Patterns

**What:** Creating new authentication mechanism for highlights.
**Why bad:** Inconsistent security model, more code to maintain.
**Instead:** Reuse existing `authDeviceMiddleware` from `/syncs/progress`.

### Anti-Pattern 4: Storing Highlights Without Book Association

**What:** Storing highlights independent of book records.
**Why bad:** Cannot display highlights in book detail view.
**Instead:** Use `koreader_partial_md5` to link highlights to books.

## Scalability Considerations

| Concern | At 100 highlights | At 10K highlights | At 1M highlights |
|---------|-------------------|-------------------|------------------|
| Database queries | Simple SELECT | Add pagination | Add cursor-based pagination |
| Sync endpoint | Store all at once | Store all at once | Consider batch processing |
| Display in UI | Show all | Paginate (20/page) | Lazy load with infinite scroll |
| Deduplication | In-memory hash | Database unique constraint | Database unique constraint |
